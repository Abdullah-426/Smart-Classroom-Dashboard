1) PROJECT INTRODUCTION
===============================================================================

What is this project?
- This is an IoT-based Smart Classroom monitoring and control system.
- It shows live classroom conditions and lets us control classroom devices remotely.
- It combines simulation (Wokwi), automation logic (Node-RED), data storage, and a modern web dashboard.

What problem does it solve?
- Avoids wastage of electricity (lights/fans/AC management).
- Gives quick visibility of temperature, occupancy, and device status.
- Supports attendance tracking using RFID cards.
- Helps in better classroom management with fewer manual steps.

Main idea in one line:
- Sensor data comes from Wokwi, logic runs in Node-RED, and results are shown/controlled from the frontend dashboard.

2) TECH STACK (CRISP + EASY WORDING)
===============================================================================

Hardware simulation side:
- Wokwi (ESP32-based virtual setup)
- Sensors: PIR motion, DHT22 temperature, RFID reader
- Output devices: light indicators, fan indicators, AC indicators, buzzer

Communication:
- Wi-Fi (inside simulation)
- MQTT broker (message passing between device and backend)

Logic and API layer:
- Node-RED (flow-based backend + automation + HTTP APIs)

Storage layer:
- Node.js storage bridge service
- Stores telemetry history, attendance records, and session summaries

Frontend:
- React + TypeScript dashboard
- Vite dev server
- Polling-based live UI updates

3) IMPLEMENTATION OVERVIEW

-----------------------------------------------------------------------------
3.1 WOKWI IMPLEMENTATION (SHORT EASY POINTERS)
-----------------------------------------------------------------------------

Components used and their function:
- ESP32: main controller in simulation.
- PIR sensor: detects movement (to know if classroom is occupied).
- DHT22: gives room temperature.
- RFID module: reads student card IDs for attendance.
- Two shift registers: expands output pins so we can control many lights/fans.
- 10 light LEDs: represent classroom tubelights.
- 6 fan LEDs: represent classroom fans.
- 2 AC LEDs: one for AC power state, one for cooling state.
- Buzzer: gives audible alert/feedback.

How Wokwi logic works:
- If motion is detected, room is treated as occupied for a short time window.
- In automatic mode, lights/fans/AC respond based on occupancy and temperature.
- In manual mode, user control from dashboard is followed.
- During off-hours schedule, power devices can be forced to stay off.
- RFID card scans are added into outgoing telemetry so attendance can be processed.

Important practical point:
- Wokwi is the device layer: it senses, acts, and reports continuously.


-----------------------------------------------------------------------------
3.2 NODE-RED FLOW SUMMARY (EASY WORDING)
-----------------------------------------------------------------------------

Node-RED does 5 main jobs:

1) Live data processing
- Receives classroom telemetry from MQTT.
- Cleans and enriches incoming data.
- Keeps latest system state ready for APIs/dashboard.

2) Control command handling
- Receives control requests (from dashboard/API).
- Converts them to device commands.
- Sends commands back to Wokwi via MQTT.

3) Automation logic
- Schedule logic: classroom follows configured active hours.
- AC assist logic: smooth cooling behavior with anti-flicker control.
- Alert logic: high temperature / after-hours motion notifications.

4) API gateway
- Exposes simple HTTP APIs used by frontend:
  - live telemetry
  - summary
  - ML/AI output
  - control endpoints
  - attendance endpoints

5) Storage integration
- Forwards telemetry and attendance updates to storage bridge.
- Attendance start/end/reset/export requests are routed through this layer.

In one sentence:
- Node-RED is the brain and traffic controller of the full system.


-----------------------------------------------------------------------------
3.3 FRONTEND DASHBOARD INTEGRATION (AND BUTTON CLICK FLOW)
-----------------------------------------------------------------------------

What frontend does:
- Shows live classroom status (temperature, occupancy, lights/fans/AC).
- Shows analytics, ML insight, AI report, and storage health.
- Provides attendance page with session controls and scan results.
- Polls backend every second to keep UI fresh.

How integration works (simple end-to-end):

A) Data display flow
1. Wokwi publishes live sensor/device data.
2. Node-RED processes it and updates latest values.
3. Storage service saves history.
4. Frontend calls APIs every second and renders updated UI cards/charts.

B) When you click a control button (example: turn fan/light/AC)
1. Frontend sends command to Node-RED API.
2. Node-RED validates and publishes command to MQTT.
3. Wokwi device receives command and applies change.
4. New telemetry is published.
5. Frontend sees updated status in next poll (quick visual confirmation).

C) Attendance flow (RFID)
1. Card is scanned in Wokwi RFID reader.
2. Scan event goes with telemetry to backend.
3. Storage logic decides: valid/duplicate/late/invalid.
4. Attendance page updates counts, recent scans, and student status.

Key viva explanation line:
- "Dashboard is not directly talking to hardware; it talks to Node-RED APIs, and Node-RED coordinates everything."

===============================================================================
5) VIVA PREP QUICK QUESTIONS (WITH READY ANSWER STYLE)
===============================================================================

Q1) Why did you use MQTT?
- It is lightweight, fast, and ideal for IoT publish/subscribe communication.
- It decouples device and dashboard systems cleanly.

Q2) Why Node-RED?
- Very suitable for IoT workflow orchestration.
- Easy to build real-time logic, API endpoints, and integration flows.

Q3) How is energy saving achieved?
- Auto control based on occupancy and temperature.
- Time-based schedule logic to avoid unnecessary usage.

Q4) How do you confirm a command really worked?
- Device publishes updated telemetry after applying command.
- Dashboard reflects actual device state in next refresh cycle.

Q5) What is the role of storage bridge?
- Persistent storage for history, sessions, attendance, and downtime stats.
- Supports trend charts and attendance records over time.

Q6) If internet or stream is unstable, what happens?
- System shows stale/disconnected indication.
- Historical and last known data handling keeps UI stable.
